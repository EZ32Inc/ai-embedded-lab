# AEL 并行设计空间探索策略 v0.1

**日期：** 2026-03-16
**类型：** 战略备忘录
**状态：** 概念阶段，尚未实现

---

## 1. 核心思想

传统的嵌入式工程师工作流是线性的：

```
想法 → 写代码 → 编译 → 烧录 → 观察 → 分析 → 修改 → 重复
```

每次只走一条路。失败了回头，换一个方向再试。

AEL 的长期优势在于：**它不必像人一样一次只做一件事。**

AEL 可以同时探索多条路径，在并行中搜索答案，而不是串行地猜测和修正。

---

## 2. 为什么这是根本性的差异

### 人类工程师的限制

- 一次只能专注一个假设
- 切换上下文有成本（重新接线、等待编译、重新分析）
- 调试时间大部分花在"等待"和"切换"上
- 经验决定走哪条路——经验不足就走弯路

### AEL 的潜力

- 多个固件变体可以并行编译
- 多块板子可以并行烧录和测试（多 instrument 环境）
- 多个假设可以同时验证，而不是逐一排除
- 结果汇总后选择最优路径，而不是靠猜测

### 一个具体的例子

**问题：** STM32F401 上 SPI 传输偶发错误，原因不明。

**人类工程师做法（串行）：**
1. 猜测是时钟速度问题 → 改代码 → 烧录 → 测试 → 无效
2. 猜测是 NSS 极性问题 → 改代码 → 烧录 → 测试 → 无效
3. 猜测是 DMA 配置问题 → 改代码 → 烧录 → 测试 → 有效
耗时：可能几小时

**AEL 并行做法：**
1. 同时生成 4 个固件变体（不同时钟速度、不同 NSS 极性、不同 DMA 配置、轮询模式）
2. 并行编译所有变体
3. 逐一烧录测试（或多板并行）
4. 汇总结果，定位有效变体
耗时：可能几分钟

## 3. 四个核心能力组件

并行探索需要以下能力组件，目前尚未全部实现：

### 组件 A：任务分解（Task Decomposition）

将一个调试目标分解为多个独立的可验证假设。

例如：
```
目标: 找出 SPI 失败原因
  ├── 假设 1: 时钟速度过高
  ├── 假设 2: NSS 极性配置错误
  ├── 假设 3: DMA 配置问题
  └── 假设 4: 轮询模式是否正常（基线对照）
```

**当前状态：** 手动分解，无工具支持。

### 组件 B：并发 Worker 编排（Worker Orchestration）

多个固件变体并行编译、并行烧录、并行测试。

需要：
- 多 instrument 支持（多块板同时运行）
- 或单 instrument 串行执行多变体（仍比手动快）
- Worker 之间的任务调度和状态追踪

**当前状态：** `verify-default repeat` 是串行的。并行 worker 框架尚未实现。

### 组件 C：结果聚合（Result Aggregation）

汇总多个变体的测试结果，自动识别：
- 哪些变体通过
- 哪些变体失败
- 成功/失败的关键差异在哪里

**当前状态：** 每个 run 独立记录 `result.json`。跨 run 比较需要手动分析。

### 组件 D：经验反馈（Experience Feedback）

将本次探索的结论反馈给下一次类似工作：
- 哪种假设方向通常有效
- 哪种变体空间值得优先探索
- 什么情况下可以缩小搜索范围

**当前状态：** Layer C Skill 文档是手动的。无自动反馈机制。

## 4. 与当前 AEL 架构的关系

当前 AEL 的执行模型是：

```
一个 project → 一个 test plan → 一次 run → 一个结果
```

并行探索需要扩展为：

```
一个 goal → 多个 hypothesis → 多个 test plan → 多个并行 run → 结果聚合 → 最优路径
```

这不需要推翻现有架构。它是在现有 run/project 模型之上增加一个"探索层"。

---

## 5. 近期可做的最小步骤

不需要等待完整框架。以下是近期就可以做的渐进步骤：

### 步骤 1（可以现在做）：多变体 test plan 批量运行

定义一个"探索包"（exploration pack），包含同一目标的多个变体 test plan，
用 `smoke pack` 的方式批量执行：

```json
{
  "name": "spi_debug_exploration",
  "board": "stm32f401rct6",
  "mode": "exploration",
  "hypothesis": "SPI 失败原因定位",
  "tests": [
    "tests/plans/stm32f401_spi_clk_slow.json",
    "tests/plans/stm32f401_spi_clk_fast.json",
    "tests/plans/stm32f401_spi_polling.json",
    "tests/plans/stm32f401_spi_dma.json"
  ]
}
```

### 步骤 2（近期）：run 结果对比命令

```bash
python3 -m ael run diff <run_id_1> <run_id_2>
```

比较两次 run 的信号特征差异，辅助定位变体之间的行为差异。

### 步骤 3（中期）：自动假设生成

给定一个失败的 run，AEL 自动生成若干个修改假设和对应的固件变体，
提交给用户审查后并行执行。

## 6. 战略意义

并行探索是 AEL 从"自动化工具"升级为"工程智能系统"的关键一步。

| 能力 | 当前 AEL | 并行探索 AEL |
|------|---------|------------|
| 调试速度 | 与人类工程师相当 | 比人类快 N 倍（N = 并行度） |
| 覆盖范围 | 受人类注意力限制 | 可系统性覆盖整个假设空间 |
| 经验利用 | 手动查阅 Skills | 自动权重化优先探索高概率路径 |
| 可扩展性 | 线性增长 | 可随 instrument/板卡数量水平扩展 |

这个方向不是今天要实现的。
但今天把它写清楚，是为了让近期的架构决策（run 模型、pack 格式、result 结构）
都朝着支持并行探索的方向演进，而不是在未来需要推倒重来。
