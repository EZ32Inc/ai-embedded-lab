# STM32F103 ADC External Input Contract v0.1

## Purpose

Define the first concrete external-input contract for a generated ADC example
without requiring immediate live execution.

Target:

- `stm32f103_adc_banner`

## Contract

The execution contract for the first bounded STM32 ADC path is:

- loopback source: `PA1`
- DUT signal: `PA0/ADC1_IN0`
- kind: `analog_in`
- physical requirement: `PA1 -> PA0`
- intended drive: about `100 Hz` square wave generated by the DUT itself

## Meaning

This contract is now explicit and formal.

It means:

- connection questions can be answered formally
- the ADC source is no longer unspecified
- the required physical setup is one bounded DUT-local loopback
- runtime validation can proceed once that jumper is present

## Current Verification Shape

The current bounded verification method is GPIO-only:

- firmware drives `PA1`
- firmware samples `PA0/ADC1_IN0`
- firmware mirrors the ADC-validated result onto `PA4`
- AEL verifies `PA4` through the existing STM32 GPIO signal path

This is a bounded ADC execution proof, not a direct external ADC waveform/value
measurement path.

## Measurement Note

For this setup, the observed `PA4` frequency from the ESP32JTAG capture path
should be treated as a coarse observed band, not as a precise frequency
measurement.

Reason:

- the ADC-validated `PA4` waveform is produced inside a software loop
- loop timing and capture timing introduce drift
- the correct pass criterion is the observed *band* that distinguishes good vs
  bad ADC loopback behavior, not an exact nominal frequency value

## Current Remaining Blocker

The remaining blocker is only whether the physical jumper is in place:

- `PA1 -> PA0`

## Why This Is The Right First ADC Step

This removes one real blocker without requiring:

- UART setup work
- broad ADC execution rollout
- broader bench redesign

It is a bounded execution step on top of the existing STM32F103 golden setup.
